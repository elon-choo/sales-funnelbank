## 1. 전체 사용자 여정 맵

### 1.1 High-Level Journey Map

```
[비회원]                    [승인 대기]                  [승인 완료 회원]
    |                           |                              |
    v                           v                              v
+--------+   +--------+   +-----------+   +--------+   +---------------+
| 랜딩   |-->| 회원   |-->| 승인 대기 |-->| 로그인 |-->| 대시보드      |
| 페이지 |   | 가입   |   | 화면      |   |        |   |               |
+--------+   +--------+   +-----------+   +--------+   +-------+-------+
                                                               |
                          +------------------------------------+
                          |
          +---------------+---------------+---------------+
          |               |               |               |
          v               v               v               v
    +-----------+   +-----------+   +-----------+   +-----------+
    | AI 기획   |   | 랜딩페이지|   | 상세페이지|   | 내 페이지 |
    | 도우미    |   | 빌더      |   | 빌더      |   | 관리      |
    +-----------+   +-----------+   +-----------+   +-----------+
          |               |               |
          v               v               v
    +-----------+   +-----------+   +-----------+
    | 프롬프트  |-->| HTML 생성 |-->| 배포/공유 |
    | 생성      |   | & 미리보기|   |           |
    +-----------+   +-----------+   +-----------+
```

### 1.2 사용자 상태 정의

| 상태 | 설명 | 접근 가능 기능 |
|------|------|---------------|
| `guest` | 비로그인 상태 | 랜딩페이지 조회, 회원가입 |
| `pending` | 가입 완료, 승인 대기 | 승인 대기 안내 화면만 |
| `approved` | 승인 완료, 활성 회원 | 모든 기능 |
| `suspended` | 이용 정지 | 정지 안내 화면만 |

### 1.3 주요 사용자 시나리오

#### 시나리오 A: 첫 방문 → 첫 랜딩페이지 완성
```
1. 마스터클래스 수료 후 웹앱 URL 안내 받음
2. 회원가입 (이메일, 비밀번호, 수강 정보)
3. 승인 대기 (평균 24시간)
4. 승인 완료 이메일 수신
5. 로그인 → 대시보드
6. "새 랜딩페이지 만들기" 클릭
7. AI 기획 도우미와 대화 (약 15-20분)
8. 프롬프트 생성 완료
9. 랜딩페이지 빌더로 이동
10. 이미지 업로드 + 생성
11. 미리보기 → 수정 → 배포
12. 고유 URL 획득 → SNS/카톡 공유
```

#### 시나리오 B: 재방문 → 기존 페이지 수정
```
1. 로그인
2. 대시보드 → "내 페이지" 탭
3. 수정할 페이지 선택
4. 편집 모드 진입
5. 텍스트/이미지 수정
6. 저장 → 자동 배포
```

#### 5.1.1 인식의 용이성 (Perceivable)

| 항목 | 체크 | 구현 방법 |
|------|------|----------|
| 대체 텍스트 | [ ] | 모든 이미지에 alt 속성 |
| 자막/대본 | [ ] | 비디오 가이드에 자막 제공 |
| 색상 대비 | [ ] | 텍스트 4.5:1, 큰 텍스트 3:1 |
| 색상 의존 금지 | [ ] | 아이콘+색상+텍스트 조합 |
| 텍스트 크기 조절 | [ ] | 200%까지 확대 가능 |

```
[색상 대비 예시]
- 본문 텍스트: #1F2937 on #FFFFFF (대비 13.5:1) OK
- 버튼 텍스트: #FFFFFF on #3B82F6 (대비 4.5:1) OK
- 에러 텍스트: #DC2626 on #FFFFFF (대비 4.5:1) OK
- 비활성 텍스트: #9CA3AF on #FFFFFF (대비 3:1) 주의
```

#### 5.1.2 운용의 용이성 (Operable)

| 항목 | 체크 | 구현 방법 |
|------|------|----------|
| 키보드 접근 | [ ] | 모든 기능 Tab/Enter로 이용 |
| 포커스 표시 | [ ] | 명확한 포커스 링 표시 |
| 건너뛰기 링크 | [ ] | "본문으로 바로가기" 링크 |
| 시간 제한 | [ ] | 자동 로그아웃 경고 및 연장 |
| 깜빡임 제한 | [ ] | 3회/초 이하 애니메이션 |

```css
/* 포커스 스타일 */
:focus-visible {
  outline: 2px solid #3B82F6;
  outline-offset: 2px;
}
```

#### 5.1.3 이해의 용이성 (Understandable)

| 항목 | 체크 | 구현 방법 |
|------|------|----------|
| 언어 명시 | [ ] | `<html lang="ko">` |
| 일관된 네비게이션 | [ ] | 모든 페이지 동일 메뉴 위치 |
| 입력 도움말 | [ ] | 레이블, placeholder, 힌트 |
| 오류 식별 | [ ] | 필드 하이라이트 + 메시지 |
| 오류 수정 제안 | [ ] | 구체적인 수정 방법 안내 |

```html
<!-- 접근성 높은 폼 필드 -->
<label for="email">이메일 *</label>
<input
  id="email"
  type="email"
  aria-required="true"
  aria-describedby="email-hint email-error"
  aria-invalid="true"
/>
<span id="email-hint" class="hint">example@email.com 형식</span>
<span id="email-error" class="error" role="alert">
  올바른 이메일 형식이 아닙니다
</span>
```

#### 5.1.4 견고성 (Robust)

| 항목 | 체크 | 구현 방법 |
|------|------|----------|
| 유효한 HTML | [ ] | W3C Validator 통과 |
| ARIA 사용 | [ ] | 역할, 상태, 속성 명시 |
| 스크린리더 테스트 | [ ] | VoiceOver, NVDA 테스트 |

### 5.2 컴포넌트별 접근성 명세

#### 5.2.1 버튼
```html
<!-- 기본 버튼 -->
<button type="button">저장하기</button>

<!-- 아이콘만 있는 버튼 -->
<button type="button" aria-label="닫기">
  <svg aria-hidden="true">...</svg>
</button>

<!-- 로딩 상태 버튼 -->
<button type="button" disabled aria-busy="true">
  <span aria-hidden="true">[스피너]</span>
  저장 중...
</button>
```

#### 5.2.2 모달
```html
<div
  role="dialog"
  aria-modal="true"
  aria-labelledby="modal-title"
  aria-describedby="modal-description"
>
  <h2 id="modal-title">페이지 삭제</h2>
  <p id="modal-description">삭제하면 복구할 수 없습니다.</p>
  <button>취소</button>
  <button>삭제</button>
</div>
```

#### 5.2.3 탭
```html
<div role="tablist" aria-label="페이지 관리">
  <button role="tab" aria-selected="true" aria-controls="panel-1">
    내 페이지
  </button>
  <button role="tab" aria-selected="false" aria-controls="panel-2">
    템플릿
  </button>
</div>
<div role="tabpanel" id="panel-1" aria-labelledby="tab-1">
  <!-- 내용 -->
</div>
```

#### 5.2.4 알림/토스트
```html
<div role="alert" aria-live="polite">
  저장되었습니다.
</div>

<div role="alert" aria-live="assertive">
  오류가 발생했습니다. 다시 시도해 주세요.
</div>
```

## 7. 에러 핸들링 매트릭스

### 7.1 에러 유형별 처리

| 에러 유형 | 사용자 메시지 | 기술 로그 | 복구 방법 |
|-----------|--------------|----------|----------|
| 네트워크 끊김 | "인터넷 연결을 확인해 주세요" | `ERR_NETWORK` | 자동 재시도 (3회) |
| 서버 오류 (500) | "일시적인 문제가 발생했습니다" | 상세 스택 트레이스 | 재시도 버튼 |
| 인증 만료 (401) | "다시 로그인해 주세요" | 토큰 만료 정보 | 로그인 리다이렉트 |
| 권한 없음 (403) | "접근 권한이 없습니다" | 권한 정보 | 홈으로 이동 |
| 리소스 없음 (404) | "페이지를 찾을 수 없습니다" | 요청 URL | 홈으로 이동 |
| 입력 오류 (400) | 필드별 구체적 메시지 | 검증 실패 필드 | 해당 필드 포커스 |
| AI 생성 실패 | "생성에 실패했습니다" | AI 응답 내용 | 재시도 또는 입력 수정 |
| 파일 업로드 실패 | "파일 업로드에 실패했습니다" | 파일 정보 | 다른 파일 시도 |
| 할당량 초과 | "이번 달 사용량을 초과했습니다" | 사용량 정보 | 업그레이드 안내 |

### 7.2 입력 검증 규칙

```yaml
이메일:
  규칙: "RFC 5322 형식"
  에러_메시지: "올바른 이메일 형식이 아닙니다"
  예시: "example@email.com"

비밀번호:
  규칙:
    - 최소 8자
    - 영문, 숫자 포함
  에러_메시지:
    - "비밀번호는 8자 이상이어야 합니다"
    - "영문과 숫자를 포함해야 합니다"

프롬프트:
  규칙:
    - 최소 50자
    - 최대 5000자
  에러_메시지:
    - "기획 정보가 너무 짧습니다 (최소 50자)"
    - "입력 가능한 최대 길이를 초과했습니다"

이미지:
  규칙:
    - 형식: JPG, PNG, WebP
    - 크기: 최대 5MB
    - 개수: 최대 10장
  에러_메시지:
    - "지원하지 않는 이미지 형식입니다"
    - "이미지 크기가 너무 큽니다 (최대 5MB)"
    - "이미지는 최대 10장까지 업로드할 수 있습니다"
```

## 8. 성능 최적화 가이드

### 8.1 로딩 성능 목표

| 지표 | 목표값 | 측정 방법 |
|------|--------|----------|
| FCP (First Contentful Paint) | < 1.5초 | Lighthouse |
| LCP (Largest Contentful Paint) | < 2.5초 | Lighthouse |
| TTI (Time to Interactive) | < 3초 | Lighthouse |
| CLS (Cumulative Layout Shift) | < 0.1 | Lighthouse |

### 8.2 이미지 최적화

```yaml
이미지_로딩:
  레이지_로딩:
    - 뷰포트 밖 이미지는 지연 로딩
    - placeholder: 흐림 처리 또는 스켈레톤

  최적화:
    - WebP 형식 우선
    - srcset으로 반응형 이미지
    - 적절한 압축률 (품질 80%)

  예상_크기:
    - 썸네일: 150x150, ~20KB
    - 미리보기: 400x600, ~80KB
    - 원본: 1200x1800, ~200KB
```

### 8.3 코드 분할

```yaml
라우트_기반_분할:
  - 대시보드 번들
  - AI 도우미 번들
  - 빌더 번들
  - 설정 번들

컴포넌트_레이지_로딩:
  - 모달 컴포넌트
  - 리치 텍스트 에디터

```
